# T020 — Undo toast for Clear all

2026-07-14

## What I did

- Added `Sources/Views/ClearToast.swift` — a soft, theme-consistent banner
  ("Cleared N items — Undo") that floats above the pinned add bar (stacked
  in `.safeAreaInset(edge: .bottom)`, never over it) rather than covering
  it. VoiceOver-announced on appear (`UIAccessibility.post(.announcement)`);
  the container carries no accessibility modifier itself (that collapses
  children into one element — see below), so the message and Undo button
  stay two separately reachable elements. The Undo button gets an explicit
  `minWidth/minHeight: 44` hit region since its visual text is small.
- Wired it into `Sources/Views/ShoppingListView.swift`: `clearGot()` now
  soft-deletes — it buffers the checked items' `persistentModelID`s in a
  new `pendingClearIDs` set (filtered out of `recentlyGot` so they vanish
  from the list immediately) and shows the toast, instead of calling
  `context.delete` right away. A token-guarded `DispatchQueue.main.asyncAfter`
  (duration from the new `TestHooks.clearToastDuration`, routed through
  `AppClock`/`TestHooks` conventions) actually deletes the buffered items
  once the toast expires (`commitClear()`), unless **Undo** is tapped first
  (`undoClear()`, which just clears the buffer — the real SwiftData rows
  were never touched, so `persistentModelID`-based state like
  `expandedID`/`flashID` can't break, per the spec's explicit "not by
  re-inserting copies" instruction). All three animate via
  `withAppAnimation`, with a Reduce-Motion-aware easing/transition choice.
- `TestHooks.clearToastDuration`: 5s in production, 3s under
  `-uiTestingDisableAnimations` (see "what I learned" below for why not
  shorter).
- New UITests in `UITests/CheckOffFlowTests.swift`:
  `testClearAllUndoRestoresItems` (check off Milk + Eggs → Clear all → tap
  Undo → both rows return with the Got-it section, counts, and identifiers
  intact) and `testClearAllExpiredToastLeavesItemsCleared` (Clear all → let
  the toast expire on its own → item stays gone). New
  `AccessibilityAuditTests.testClearAllUndoToastPassesAccessibilityAudit`
  runs `performAccessibilityAudit` with the toast on screen.
- README: documented the undo toast under the existing "Clear all" bullet.

## What I learned debugging (kept in case this recurs)

Two non-obvious SwiftUI accessibility pitfalls cost most of the iteration
time here, both confirmed by exporting `.xcresult` screenshots/hierarchy
dumps and a throwaway instrumented test (see git history if resurrecting):

1. **Any accessibility modifier applied directly to a container view (no
   explicit `.accessibilityElement(children:)`) can silently collapse its
   children into one element.** The toast's outer `HStack` originally had
   `.accessibilityIdentifier("clearToast.banner")` on it; that made the
   Undo `Button` disappear from the accessibility tree entirely (not just
   mis-identified — genuinely absent, confirmed via
   `app.descendants(matching: .any)`). Fix: no accessibility modifier on
   the container; identifiers live only on the leaf `Text`/`Button`.
2. **XCUITest's own synchronization overhead (each `waitFor…`/tap can cost
   the better part of a second) eats into a short test-mode timer.** An
   initial `clearToastDuration` of 1.2s under `disableAnimations` expired
   mid-test, before the undo-in-time flow even got to check for the
   button — not a logic bug, just too tight a budget. 3s gives comfortable
   margin for both the undo-in-time and let-it-expire flows.
