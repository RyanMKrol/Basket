# T040 — Fix ClearedCelebration's clip-out dismiss

## 2026-07-16

### What I did

Two coordinated changes, per the spec (a curve-only tweak already failed once —
T002 / d112d01 — so this deliberately does not stop at re-tuning an easing curve):

1. **`Sources/Views/ClearedCelebration.swift`**: added `var isDismissing: Bool =
   false` and an `.onChange(of: isDismissing)` that animates `shown` back to
   `false` (`.easeOut(duration: reduceMotion ? 0.3 : 0.4)`) whenever the caller
   sets it — so the 🎉's `scaleEffect`/`opacity` and the "All done!" label's
   `opacity` animate back down themselves, instead of staying pinned at
   full scale/opacity for the whole dismiss while only the outer container faded.

2. **`Sources/Views/ShoppingListView.swift`**: consolidated celebration dismissal
   into one path:
   - Added `@State private var celebrationDismissing = false` and a new
     `dismissCelebration()` that (a) flips `celebrationDismissing = true` inside
     `withAppAnimation` — which `ClearedCelebration` picks up via `isDismissing`
     and starts fading its own content — then (b) after
     `celebrationExitDuration` (0.4s, 0.3s under Reduce Motion; 0.05s when
     `TestHooks.disableAnimations`) unmounts the overlay by setting
     `celebrating = false`.
   - `celebrateCleared()`'s scheduled auto-dismiss now calls
     `dismissCelebration()` instead of inlining its own `celebrating = false`.
   - The overlay's mount condition (`body`, ~line 110) dropped the
     `!listIsEmpty` half of the old `celebrating && !listIsEmpty` compound
     condition — now gated on `celebrating` alone, so there's exactly one thing
     that ends the celebration. The old `!listIsEmpty` guard's actual intent
     (never overlap the celebration with the empty state) is preserved by
     instead deferring `EmptyStateView` itself: `listIsEmpty && !celebrating`
     (~line 74), so the empty state can't render underneath/behind a still-
     visible celebration.
   - `clearGot()` now calls `dismissCelebration()` when `celebrating` is true,
     right after buffering the clear — so emptying "Got it" mid-celebration
     (e.g. via "Clear all") routes through the *same* graceful dismiss instead
     of the list emptying out from under the overlay via a second,
     uncoordinated trigger (the actual T002-adjacent bug: two independent ways
     to end the same view).
   - Did not touch `Sources/Views/SparkleBurst.swift` (out of scope; its
     one-shot burst completes long before the celebration's dismiss window).

### Mandatory recorded-video verification (not code reasoning alone)

Added `UITests/CheckOffFlowTests.swift ::
testCelebrationDismissFadesGracefullyOnClearGot` (`realTiming: true`, so
production animation curves/durations run, not the sped-up test defaults).
Note: `TestHooks.celebrationDuration` is unconditionally `nil` whenever
`-uiTesting` is passed (by design, so the transient auto-dismiss can't race a
`waitForExistence` poll — see the existing comment on
`testClearingWholeListShowsCelebration`), which means the *timed* auto-dismiss
never fires under any UI test, `realTiming` or not. The only dismiss trigger
reachable under test is therefore the same one a real user has: tapping
"Clear all" in the Got it section while the celebration is showing — which,
after this fix, is routed through the identical `dismissCelebration()` path as
the timed auto-dismiss (see change 2 above). So this one recording verifies
both spec item 3's "graceful fade" requirement and item 3d's "compound-
condition hazard" reproduction at once, since post-fix they are literally the
same code path — there is no longer a second, differently-behaved trigger to
record separately.
