## T051 — Dynamic Type: thread relativeTo through Theme's font builders + a scaling UI test

**2026-07-17**

### What I did

- `Sources/Theme/Theme.swift`: added a `relativeTo textStyle: Font.TextStyle` parameter to
  `Theme.body` (default `.body`) and `Theme.title` (default `.title`), threaded through the
  private `font(...)` builder. The `.custom` branch now calls
  `Font.custom(name, size: size * scale, relativeTo: textStyle)`. All 31 existing call sites
  keep compiling unchanged via the defaults. The unused `.rounded`/`.monospaced` branches are
  left building plain `.system(size:weight:design:)` with a comment noting they don't
  participate in Dynamic Type yet (pastelDots, the app's only theme, uses `.custom` for both
  fonts) — revisit if a non-custom theme is ever added.
- Added `UITests/DynamicTypeTests.swift` (`testItemNameScalesUnderAccessibilityContentSize`):
  launches the app normally, waits for the Milk row's name label
  (`A11yID.ItemRow.nameLabel("Milk")`, queried as a button per house rule) via a bounded wait,
  records its frame height; relaunches with
  `-UIPreferredContentSizeCategoryName UICTContentSizeCategoryAccessibilityM` appended, waits
  for the same element, and asserts its frame height is strictly greater. Attaches a
  screenshot at each stage.
- `README.md`: updated the Pastel Dots intro paragraph to describe the new
  `relativeTo:`-based scaling and the `.rounded`/`.monospaced` limitation; updated the
  `AccessibilityAuditTests.swift` bullet's `.dynamicType` exclusion rationale (no longer
  "fixed-size fonts" — now "layout fallout not yet audited, revisited in a follow-up unit")
  and added a bullet for the new `DynamicTypeTests.swift`.
- Did NOT touch `UITests/AccessibilityAuditTests.swift` or `.harness/` files, per scope.

### Test-first check

Before finalizing, I stashed the Theme.swift change and ran the new UI test against the
unmodified (two-arg `.custom(name, size:)`) code to confirm it discriminates the right thing.
It surprisingly PASSED even without `relativeTo:` (measured heights: 20pt default → 30pt at
`AccessibilityM`), because `Font.custom(_:size:)` on this SDK already scales implicitly
relative to `.body` — the same curve `Theme.body`'s new explicit default now names. So the
behavioural delta this task adds is: (a) `Theme.title` now scales relative to `.title`'s curve
instead of implicitly whatever the two-arg initializer defaulted to, and (b) call sites can now
opt into a more precise text style (e.g. `.caption`). The prescribed test (spec step 3) still
exercises real, non-trivial Dynamic Type scaling end-to-end on the shipped code path and passes
with the change applied; I verified this is genuine layout scaling (not a fixed accessibility
hit-target inflation) by checking actual measured heights, not just a boolean assertion.
