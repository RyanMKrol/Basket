## T052 — Re-enable the `.dynamicType` accessibility audit, fix what it flags

**2026-07-17**

### What I did

- `UITests/AccessibilityAuditTests.swift`: removed `.dynamicType` from the excluded-category set
  and its now-stale "pixel fonts are fixed-size by design" doc bullet (that stopped being true
  once T051 threaded `relativeTo:` through `Theme`'s font builders). Updated the class-level
  comment.
- Ran the five audit tests and iterated on every issue raised:
  - **Decorative emoji `Text` at a fixed `.system(size:)`** (AboutView's basket/tip-badge emoji,
    EmptyStateView's basket/holiday-accent emoji, AddBar's suggestion-row emoji,
    ClearedCelebration's 🎉, LaunchFlourish's 🧺) were flagged as fully "unsupported" —
    `accessibilityHidden(true)` does **not** exempt an element from this audit (verified
    empirically: it still inspects rendered layout, not just the accessibility tree). Fixed each
    with `@ScaledMetric(relativeTo:)` seeded at the original literal size, so they scale like any
    other Dynamic Type element.
  - **Exception:** `ItemRow`'s row emoji sits in a *fixed* 34x34 frame. I initially scaled it too,
    but visual verification at `UICTContentSizeCategoryAccessibilityXXXL` showed the glyph
    overflowing that fixed box (the frame doesn't grow with it) — reverted to a plain fixed
    24pt size with a comment explaining why, since it was never flagged by the audit in the
    first place (a decorative pictograph inside a fixed box, not readable text).
  - **Two elements the audit flagged as "partially unsupported"** even with `relativeTo:` and
    `.fixedSize(vertical: true)` applied — `EmptyStateView`'s rotating title line and
    `QuantityEditor`'s value display — turned out to be a **static** heuristic (confirmed by
    testing: neither `fixedSize` nor a `.dynamicTypeSize` cap ever changed the audit's reported
    frame, so it isn't re-rendering at other categories, just predicting overflow from the
    current layout). Both are genuine layout risks per the task's own framing (a fixed-width
    single line; a stepper/value/clear-button row with no slack) — capped at
    `.dynamicTypeSize(...DynamicTypeSize.accessibility2)` per the task's explicit authorization,
    and the resulting (expected, unavoidable) audit findings suppressed narrowly by
    accessibility identifier in `handleIssue`, mirroring the existing `.elementDetection`
    exception pattern.
  - Visual verification at the largest possible category (`...AccessibilityXXXL`, via a new
    `DynamicTypeTests.testLargestAccessibilitySizeScreensRenderWithoutBreaking`) additionally
    surfaced a real overflow in `AboutView`'s tip badges (label + price outgrowing a fixed
    82x86 box, visibly spilling past the rounded background) and truncated (not just fixed-size)
    subtitle/tip-prompt text (squeezed by the sheet's `.medium`/`.large` detents) — neither was
    caught by the automated audit itself. Fixed the badge frame to `.frame(width: 82)` +
    `.frame(minHeight: 86)` (so it can grow) and capped all three (badge label, badge price,
    subtitle, tip prompt) at `.accessibility2`, again suppressed narrowly by identifier.
- `README.md`: updated the audit section (only `.contrast`/`.textClipped` are now wholesale
  category exclusions) and added a "Dynamic Type note" enumerating every capped element and why,
  plus a mention of the new large-size screenshot test.
