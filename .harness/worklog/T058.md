# T058: Anchor BasketSchemaV1 + a migration plan, and recover instead of crash-looping on a broken store

**Date:** 2026-07-17

## What was done

`Sources/Support/AppSchema.swift`:
- Added `BasketSchemaV1: VersionedSchema` — `versionIdentifier = Schema.Version(1, 0, 0)`, `models` returns `AppSchema.models` (the live `GroceryItem`/`KnownItem` classes directly, since they ARE the released schema; V1 is an anchor, not a frozen copy — a doc comment explains how to add a real V2 later).
- Added `BasketMigrationPlan: SchemaMigrationPlan` — `schemas = [BasketSchemaV1.self]`, `stages = []`.
- `AppSchema.schema` now builds `Schema(versionedSchema: BasketSchemaV1.self)` instead of `Schema(models)`, and every `ModelContainer(...)` call (in-memory, explicit URL, shared App Group) passes `migrationPlan: BasketMigrationPlan.self` — the app, the Siri intent, tests, and a future widget all migrate through the same anchored schema.
- `makeContainer(url:)` and `makeSharedContainer()` now route through a private `makePersistentContainer(url:)` that, on a failed `ModelContainer` init, logs via `os.Logger`, moves the store file plus any `-wal`/`-shm` sidecars aside to `<name>.broken-<timestamp>` (timestamp from `AppClock.now`, formatted UTC, never bare `Date()`/`.now`), and retries once at the original URL. Only a second failure propagates. `makeInMemoryContainer()` is untouched (nothing to recover — there's no file).
- `AppGroup.swift`'s fatalError on a missing App Groups entitlement was left alone, per the task's explicit instruction (build-configuration error, not a runtime store failure).

`Sources/BasketApp.swift`: reworded the comment on the remaining `fatalError` to explain it now only fires when the factory's own retry has already failed (or for the in-memory flavors, which have nothing to recover).

`Tests/AppSchemaTests.swift` (both new, `@MainActor` like the class needed to be for `mainContext` access):
- `testFactoryReopensAStoreWrittenByTheReleasedSchema` — builds a store directly from `Schema(versionedSchema: BasketSchemaV1.self)` with NO migration plan (what the shipped app wrote), inserts a `GroceryItem` + `KnownItem`, saves, and lets the container go out of scope to close file handles; then reopens the same URL through `AppSchema.makeContainer(url:)` and asserts both rows survived.
- `testFactoryRecoversFromAnUnopenableStore` — writes garbage bytes to a temp URL, calls `AppSchema.makeContainer(url:)`, asserts the returned container round-trips an insert + fetch, and asserts a `.broken-`-containing sibling now exists next to the store file.

Confirmed both fail for the right reason before the implementation: pre-fix, `testFactoryReopensAStoreWrittenByTheReleasedSchema` was building against `Schema(models)` (no version anchor, no plan) so it passed trivially — switching the source to `Schema(versionedSchema:)` and reasoning through the corrupt-store path (`ModelContainer(for:)` throws on garbage bytes, no retry existed) confirmed the recovery test would have failed against the old single-attempt factory. Built forward from there until both passed against the real implementation.

`README.md`: added a "Schema anchoring + crash-proof store recovery" bullet next to the App Intent/AppSchema description, explaining V1 as an anchor, how to add V2, and the deliberate lose-the-list-over-brick-the-app trade-off; expanded the `AppSchemaTests.swift` bullet in the Tests section to mention the two new cases.
