# T005 ‚Äî tap blank space to close quantity editor

2026-07-04

## What I did

Added a `.onTapGesture` to the full-bleed `BasketBackground()` instance in
`ShoppingListView.body` (the bottom layer of the outer `ZStack`). On tap it
closes the currently-open row editor by setting `expandedID = nil` inside the
same `withAnimation(.spring(response: 0.35, dampingFraction: 0.82))` block used
by `toggleQuantityEditor` / `clearQuantity`, guarded to no-op when `expandedID`
is already nil. No other files touched ‚Äî `ItemRow`'s own `.onTapGesture` and the
header/add-bar controls sit in front of the background in the view hierarchy,
so SwiftUI still resolves taps on rows/controls to their own gestures first;
only genuinely blank space reaches the new background gesture.

## What remains

Manual on-device confirmation of the tap-to-close interaction itself (open a
row's qty editor, tap blank space below the list, confirm it collapses) was
**not performed live** in this session: this build machine runs several
concurrent unattended Claude sessions sharing one physical desktop, and driving
the Simulator would require posting real mouse events to the shared screen ‚Äî
too risky (could misclick into another session's terminal). I verified instead
via build success + code review: the added gesture is byte-for-byte the same
`withAnimation(...) { expandedID = nil }` pattern already exercised by
`toggleQuantityEditor` and `clearQuantity` elsewhere in this file, guarded by
the same nil-check idiom, so its runtime behavior is not new ‚Äî only the
trigger (a background tap) is. A human should do a quick visual pass in Xcode
or on a personal machine to confirm the animation feel before considering this
fully signed off.
