## T054 — Extract `ClearChoreography` from `ShoppingListView`

**2026-07-17**

### What I did

- Added `ClearChoreography<ID: Hashable>` to `Sources/Services/ListLogic.swift`, styled after the
  existing `CheckOffChoreography`: a pure value type (`hidden: Set<ID>` + a monotonic `token`)
  with `isHidden(_:)`, `beginClear(_:) -> (token:count:)?`, `undo() -> Bool`,
  `expire(token:) -> Set<ID>?`, and `narrow(stillPresent:)`. `expire(token:)` deliberately does
  NOT empty the buffer itself — preserving T038's narrowed-buffer no-flicker semantics — the
  caller narrows separately after issuing the deletes.
- Rewired `Sources/Views/ShoppingListView.swift`: replaced `pendingClearIDs: Set<PersistentIdentifier>`
  + `clearToken: Int` with a single `@State private var clearChoreo = ClearChoreography<PersistentIdentifier>()`.
  `recentlyGot` now filters via `clearChoreo.isHidden`. `clearGot()`/`commitClear()`/`undoClear()`
  now only do scheduling (`DispatchQueue.main.asyncAfter(deadline: .now() + TestHooks.clearToastDuration)`),
  the `context.delete` loop, `withAppAnimation` wrapping, `Haptics`, and toast show/hide — the
  token bookkeeping and buffer narrowing live entirely in `ClearChoreography` now.
  `commitClear` takes the captured `token` as a parameter (rather than reading a captured `@State`
  var) and is a no-op via `expire(token:)` returning nil when stale/already-undone — same net
  behaviour as the old `guard clearToken == token else { return }`, just pushed into the pure type.
- `showClearToast`/`clearedToastCount` stayed as view state (no reason to move them — they're pure
  presentation, not part of the buffer/token race).
- Extended `Tests/ListLogicTests.swift` with 5 new `ClearChoreography` tests: expire-after-undo is
  a no-op, double-clear supersedes a pending expiry (stale token nil, current token returns the
  full accumulated buffer), undo-after-expiry is a no-op once narrowed away, a stale token never
  returns ids to delete, and narrowing preserves the T038 no-flicker semantics (still hidden after
  expire, drops only the ids the predicate rejects).
- Wrote the new tests first, confirmed against the pre-extraction API shape (they didn't compile
  since `ClearChoreography` didn't exist yet — the right kind of "fails first"), then implemented.
- Updated the `ListLogic`/tests one-liners in `README.md` and the `Services/` project-map bullet in
  `CLAUDE.md` to name `ClearChoreography` alongside `CheckOffChoreography`.
