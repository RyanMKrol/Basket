## A real bug found and fixed along the way

First full-suite run: only the three new `BasketWidgetSnapshotBuilderTests`
crashed (`EXC_BREAKPOINT`/`SIGTRAP`, "Test crashed with signal trap"),
100% reproducible, regardless of cold vs. warm simulator state, class name,
or run order (verified with an isolated minimal repro class and by running
an existing passing test class immediately beforehand in the same process —
still crashed on cue). Root-caused to **two separate bugs**, fixed together:

1. **App bug (real, not test-only):** `BasketApp`'s scenePhase `.background`
   handler now calls the real `WidgetCenter.shared.reloadTimelines(ofKind:)`.
   A unit-test host (`BasketTests`) launches the real `BasketApp`, including
   this handler — and calling the live WidgetKit API from that context, on a
   simulator that just got a freshly (re)built widget extension whose kind
   isn't registered with the system yet, reliably traps. Fixed by gating
   `WidgetReload.defaultReloadTimelines` behind `TestHooks.isHostedByXCTest`,
   the same pattern `BasketApp.init` already uses to keep the real App Group
   store out of unit tests.
2. **Test-file bug (this task's own new test):** `BasketWidgetSnapshotBuilderTests`'s
   first draft had a `makeContext() throws -> ModelContext` helper that
   returned only `container.mainContext`, discarding the `ModelContainer`.
   Once the container went out of scope its backing in-memory store could be
   torn down while the returned `ModelContext` still held a dangling
   reference — `context.fetch(...)` then trapped inside SwiftData. Every
   existing SwiftData test in this repo (`ModelTests`, `AppSchemaTests`)
   already keeps the `ModelContainer` alive for the whole test; rewrote this
   test file to match that convention (`let container = try makeContainer()`,
   `container.mainContext`) and the crash disappeared. This is why the
   full-suite failure reproduced with a throwaway probe test (bare fetch, no
   widget code at all) using the same broken pattern, and why fixing bug (1)
   alone wasn't sufficient — both bugs were live at once in the same test run.

Both fixes verified: full suite green (143 → 146 tests, all passing) after
each fix landed, confirmed by rerunning from a fresh `xcodegen generate` +
cold simulator state at least twice.

## Manual visual check — NOT performed, documented per spec's own note

The spec explicitly anticipates this: *"the app-screenshot visual-verify
hook does not capture widgets, so rendering is eyeballed by hand in the
simulator — record that check in the worklog."* Tried to find a headless
path anyway (`xcrun simctl listapps`/`launch`/`openurl` with several
undocumented widget-gallery URL schemes) — confirmed there is no public or
private `simctl` verb, launchable process, or URL scheme for Home Screen
widget-gallery navigation or placement; it's SpringBoard drag-and-drop only.
Per this repo's own convention (never drive real mouse/keyboard/screen to
verify UI — see `Tests/TipJarTests.swift`'s identical scope note for
StoreKit purchases, which are also human/Xcode-only), this specific check
could not be automated and needs a human to: build+run in the simulator,
long-press the Home Screen → "+" → search "Basket" → add the small widget,
repeat for medium, and confirm — cream/off-white Pastel Dots background,
dark ink text, "N to get" + emoji/name rows (or "All done" 🧺 when empty).
