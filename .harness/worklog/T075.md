# T075 — Swift 6 strict-concurrency migration

Completed BY HAND (owner, interactive session) after the loop blocked on it 4× (and its
ancestors T044/T070 before it). Status flipped to `done` in the same commit as the work.

## Why the loop couldn't do it

Every loop attempt at T075 (and the T070 spec it inherited) tried to make the pure-logic
service enums `@MainActor`, which forced a `@main` entry point onto the native harness
`tools/main.swift`. But the `LOCAL_DOD` gate in `harness.env` compiles that harness with a
plain `swiftc` (no `-parse-as-library`), and the builder cannot edit `.harness/` — so
`@main` + that gate = `'main' attribute cannot be used in a module that contains top-level
code`, on every attempt, unfixable from inside the task's scope.

## What was actually done (a better approach)

The pure-logic services never needed `@MainActor` (none touch UIKit/SwiftData). So:

- `Emoji.cache`, `SemanticEmoji.embedding` → `nonisolated(unsafe) static let` + justification
  (thread-safe `NSCache`; read-only single-threaded `NLEmbedding`). Services stay non-isolated.
- `tools/main.swift` stays a plain top-level script — NO `@main`, NO `-parse-as-library`.
  The only fix: the two helpers touching top-level mutable vars (`check`→`failures`,
  `seasonDate`→`utc`) are marked `@MainActor` to match Swift 6's implicitly-main-actor
  top-level context.
- App-side: `Haptics` + `QuantityController` → `@MainActor`; AppIntents statics → `static let`;
  `containerOverride`/`WidgetReload.reloadTimelines`/`LaunchOnce.fired` → `nonisolated(unsafe)`
  + justification (all documented in README's new Concurrency section).
- `UITests/BasketUITestCase` → `@MainActor` (XCUIApplication init is main-actor in Swift 6).
- `project.yml` → `SWIFT_VERSION: "6.0"`. ci.yml + README + harness.env LOCAL_DOD native-harness
  `swiftc` all get `-swift-version 6` (no `-parse-as-library` needed).
