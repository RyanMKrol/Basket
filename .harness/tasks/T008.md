## Overview

Fix ~20 semantically wrong emoji mappings found by an empirical audit (marmalade → 🍓, chicken
nuggets → 🍱, ginger → 🥔, …) in the generator's data.

## Do

In `tools/gen_emoji.py`'s RAW data (and `tools/emoji_supplement.txt` where noted), correct these
mappings — keyword → new emoji:
`marmalade`→🍊, `lemon curd`→🍋, `chicken nugget`→🍗, `ginger`→🫚, `galangal`→🫚, `turmeric`→🫚,
`assam`→🍵, `periwinkle`→🦪, `kebab`→🥙, `comb`→🪮, `hair brush`→🪮, `nail`→💅,
`birthday card`→💌, `squash`→🎃, `butternut`→🎃, `ackee`→🥭, `mirin`→🍶.
For `mirin` and `ackee`: the curated RAW entry currently wins first-seen dedupe over a better
supplement entry — remove the stale RAW duplicate so the supplement's glyph lands (or set the RAW
value directly). Also delete the other duplicate RAW keywords the dedupe pass currently masks
(`congee`, `injera`, `edamame`, `plantain`, `cassava`, `roti`, `onigiri` each appear twice), and
make the generator emit a warning when a duplicate keyword is found in its input data.
Then run the full emoji pipeline (CLAUDE.md → "Emoji pipeline"): regenerate `EmojiTable.swift`,
regenerate `SuggestionDictionary.swift` via `python3 tools/gen_suggestions.py` if the keyword set
changed, and re-run the coverage audit (`tools/audit_coverage.swift` over `tools/corpus/*.txt`) —
coverage must stay at 0 fall-throughs.

## Done when

- Each keyword above resolves to its new emoji through the real cascade (spot-check via the native
  harness or a scratch compile of `EmojiTable.match`); 🫚/🪮 render on iOS 16.4+ (already true for
  🫛/🫘 in the table).
- `gen_emoji.py` warns on duplicate keywords and its RAW data contains none.
- Coverage audit reports 0 fall-throughs; native harness + full simulator suite pass; generated
  files were regenerated, never hand-edited.
