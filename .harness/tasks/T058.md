## Overview
Basket is LIVE on the App Store with no VersionedSchema anchor — users' installed stores are the released schema forever, and any future model change becomes an unplanned migration that the simulator hides (store deleted on rebuild) but real devices hit cold. Worse, a failed container open is currently `fatalError` (Sources/BasketApp.swift:50), i.e. a failed future migration = a permanent crash loop. Anchor the shipped schema as BasketSchemaV1 + a one-version SchemaMigrationPlan threaded through the shared container factory (from the AppSchema/factory task), and replace the fatalError with a move-the-store-aside-and-recreate recovery so the app can never crash-loop on its own data.

## Do
(1) Schema anchor: define `BasketSchemaV1: VersionedSchema` (in Sources/Support/ or Sources/Models/) with `versionIdentifier = Schema.Version(1, 0, 0)` and `models` returning AppSchema.models' types — referencing the LIVE GroceryItem/KnownItem classes directly, because the current classes ARE the released schema (V1 is an anchor for future migrations, not a frozen copy; copies begin at V2 when a real change happens). Define `BasketMigrationPlan: SchemaMigrationPlan` with `schemas = [BasketSchemaV1.self]` and `stages = []`. Thread both through the shared container factory in Sources/Support/AppSchema.swift: every flavor builds `Schema(versionedSchema: BasketSchemaV1.self)` and passes `migrationPlan: BasketMigrationPlan.self` to ModelContainer, so the app, the Siri intent, tests, and the future widget all open the store through the same anchored schema. (2) Recovery: in the factory's persistent-URL paths (makeContainer(url:) / makeSharedContainer()), when ModelContainer init throws: log the error via os.Logger, move the store file AND its -wal/-shm sidecars aside to `<name>.broken-<timestamp>` (timestamp formatted from `AppClock.now` — never bare Date()/.now, per the repo's determinism rule), then retry creating a fresh container at the original URL; only if the retry ALSO throws does the error propagate (BasketApp's remaining fatalError then covers only that truly unrecoverable case). Losing a grocery list beats a bricked app — say so in a code comment. Do NOT touch AppGroup.swift's fatalError at Sources/Support/AppGroup.swift:21 — a missing App Groups entitlement is a build-configuration error, not a runtime store failure, and recovering from it would mask broken builds. Repo rules that bind this task: unit tests run against in-memory containers and temp-directory URLs only (never the real App Group store); the UITests' `-uiTesting` (in-memory) and `UITEST_STORE_URL` (temp persistent store) branches must keep working through the factory; the App Intents target compiles the same files (AddToBasketIntent.swift resolves its container via the factory after the predecessor task), so no iOS-app-only API in the schema/plan code beyond what already compiles there.

## Done when
Two new unit tests in Tests/ pass. (a) Migration-anchor test — the fixture is PRODUCED IN THE TEST, not checked in: create a store at a temp URL via a ModelContainer built from `Schema(versionedSchema: BasketSchemaV1.self)` WITHOUT the migration plan (byte-compatible with what the released app writes, since V1's models are the released classes), insert a couple of GroceryItem/KnownItem rows, save, and release the container/context (scope exit) so file handles close; then reopen the SAME URL through the factory (with BasketMigrationPlan) and assert the rows survive intact. Additionally: `grep -rn "ModelContainer(" Sources/` shows containers created ONLY inside the factory; every factory flavor passes the migration plan; the full suite passes (`xcodegen generate && xcodebuild test -project Basket.xcodeproj -scheme Basket -destination 'platform=iOS Simulator,name=iPhone 17 Pro'`) including the UITest relaunch/persistence flows; README.md documents the schema anchoring (V1 = released schema; how to add V2 later) and the recovery behavior's deliberate data-loss trade-off in the same commit.
