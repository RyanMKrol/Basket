## Overview
When the add bar is focused (keyboard open), its suggestion rows currently overlap the item list behind them with no visual separation, which reads as jarring (the "Salad / Chicken / Basket" suggestions sit directly on top of the list items). Add an animated background blur so the add bar + suggestions read as the foreground and the item list recedes into the background — built with THREE selectable blur intensities behind a temporary switch so the owner can compare all three live and pick the best.

## Do
In `Sources/Views/ShoppingListView.swift`: the add-bar focus state is `@FocusState private var addBarFocused: Bool` (~line 22); the main content (the `BASKET` header + the `ScrollView` of item rows) is the `VStack` inside the top-level `ZStack` (~lines 70–120); the `AddBar` (text field + its suggestion rows) is hosted in `.safeAreaInset(edge: .bottom)` (~line 121) and therefore renders ABOVE that content.

When `addBarFocused` is `true`, apply an animated blur (and, if it helps separation, a very subtle dim/scrim) to the MAIN CONTENT LAYER ONLY — the header + scrolling item list — while leaving the `AddBar` (field + suggestion rows) sharp in the foreground. Use SwiftUI's `.blur(radius:)` and/or a `Material` overlay. Do NOT use Liquid Glass (`.glassEffect`) — it is the wrong tool for a full-background blur (owner-confirmed).

Introduce a small `BlurStyle` enum (place it under `Sources/Views/`) with THREE intensity levels — light / medium / heavy (pick sensible starting values, e.g. radii ~3 / ~8 / ~16 pt, or `.ultraThinMaterial` / `.thinMaterial` / `.regularMaterial`) — plus a TEMPORARY in-app switch to cycle between them at runtime (e.g. a small debug segmented control shown only while the add bar is focused, driven by an `@State`) so the owner can compare all three live. This switch and the two unused levels are temporary scaffolding; a follow-up task (the hardcode step) removes them once the owner picks a winner.

Animate the blur in/out with `withAppAnimation` / `.unlessUITesting` (never a bare `withAnimation`), so the effect is disabled under UI testing. Keep the new type under `Sources/Views/` (do not spill into other directories).

## Done when
- Focusing the add bar (keyboard open) visibly blurs the background content (header + item list) while the add bar + suggestion rows stay sharp in front; the blur/animation is disabled under UI testing so the existing XCUITests still pass unchanged.
- All THREE blur intensities are reachable at runtime via the temporary switch, and switching re-renders the blur live.
- `./build_run.sh` reaches `** BUILD SUCCEEDED **` and the full simulator suite is green (no existing add / suggestion / check-off / empty-state UITest regresses).
- Visual verification (required — this is a visual change): capture the focused state at EACH of the three blur levels and record in the worklog which screenshot is which, so the owner can compare and pick in the paired review task.
