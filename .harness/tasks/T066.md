## Overview
Once tools/verify_voiceover_order.sh exists (dependency unit), builders and auditors of accessibility-affecting tasks should be told to run it - otherwise the check exists but nothing routes harness work through it. Edits .harness/custom/ overlay files, so gate: needs-human; ordered after the xcui doc unit so the two same-file edits don't collide.

## Accessibility-affecting changes: run the VoiceOver order check

If your change touches accessibility labels/hints/traits or the structure of rows, the add bar, or the quantity editor, run `./tools/verify_voiceover_order.sh` - it launches the seeded deterministic app and diffs the computed VoiceOver announcement sequence against the committed `tools/voiceover_expected.json`, failing on any focus-order or phrasing drift that `performAccessibilityAudit` cannot catch. If your task INTENTIONALLY changes announcements, re-record with `./tools/verify_voiceover_order.sh --record`, commit the new expected file, and justify the diff announcement-by-announcement in the worklog.

---

Append to `.harness/custom/visual-verify-audit.md`:

---

If the task touched accessibility labels, hints, traits, or row/add-bar/editor structure, run `./tools/verify_voiceover_order.sh` yourself: a mismatch the worklog doesn't explain, or a re-recorded `tools/voiceover_expected.json` whose diff the worklog doesn't justify announcement-by-announcement, is a FAIL.

---

Then mark done via `.harness/scripts/mark-done.sh <task-id>`.
