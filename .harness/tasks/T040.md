## Overview
The full-screen 'All done!' celebration (Sources/Views/ClearedCelebration.swift, shown from Sources/Views/ShoppingListView.swift) fades in nicely when the last to-get item is checked off, but on dismiss it visibly 'clips out' instead of fading gracefully — a real, user-reported bug, and a REPEAT: a previous fix attempt (T002, commit 496411d) only swapped the dismiss animation's easing curve (.easeIn -> .easeOut) and was explicitly REJECTED by the user (commit d112d01) with the quote 'it still clips out. It doesn't actually fade out like I asked for.' The T002 worklog admits it verified the change purely by reasoning about the curve shape in code, never by watching the animation play — do not repeat that mistake here.

## Do
Fix the ClearedCelebration dismiss animation so it fades out gracefully instead of clipping/cutting out. This requires TWO coordinated changes — a curve-only tweak already failed once (T002/d112d01), so do not stop at re-tuning the easing:

1. **Give ClearedCelebration a mirrored exit animation for its inner content.** In Sources/Views/ClearedCelebration.swift, the view has its own `@State private var shown = false`, set true exactly once in `.onAppear` (drives `.scaleEffect(shown ? 1 : 0.4).opacity(shown ? 1 : 0)` on the emoji Text and `.opacity(shown ? 1 : 0)` on the 'All done!' label) — but `shown` is NEVER reset to false, so the inner content stays pinned at full scale/opacity for the whole dismiss while only the OUTER container's `.transition(.opacity)` (in ShoppingListView.swift) ramps down around it. Add a way for ClearedCelebration to animate its OWN inner content back out (shown -> false, fading + scaling down) in step with the outer dismiss, instead of relying solely on the outer opacity transition. A natural approach: accept a binding/flag (e.g. `isDismissing: Bool` or drive off `.onChange` of a passed-in `celebrating`-like signal) so when the caller starts dismissing, ClearedCelebration itself animates `shown` back to false with a similar (but not necessarily identical) curve/duration to its entrance, so the 🎉 visibly shrinks and softens away rather than sitting frozen while the background fades.

2. **Decouple the compound `celebrating && !listIsEmpty` visibility condition** at ShoppingListView.swift line 110 (`if celebrating && !listIsEmpty { ClearedCelebration(...).transition(.opacity) }`). `listIsEmpty` (line 65: `toGet.isEmpty && recentlyGot.isEmpty`) can flip to true WHILE the celebration is still showing — most plausibly via the user tapping 'Clear got items' (clearGot(), ~line 383) during the celebration's ~1.6s on-screen window, which immediately buffers the just-checked item into `pendingClearIDs`, emptying `recentlyGot` outside of the celebration's own dismiss animation context. Today this can yank ClearedCelebration out of the view tree via the `!listIsEmpty` half of the AND, independent of and not synchronized with the celebration's own `withAppAnimation(.easeIn...) { celebrating = false }` dismiss path — a second, uncoordinated trigger for the same view's removal is exactly the kind of thing that reads as an abrupt clip. Restructure so there is exactly ONE dismiss path: e.g. gate visibility on `celebrating` alone, and make `clearGot()` (when celebrating is true) trigger the SAME graceful celebration-dismiss routine instead of independently flipping `listIsEmpty` out from under it — or snapshot/latch the emptiness state once the celebration starts so it can't be affected by a `recentlyGot` mutation mid-celebration. Preserve the existing intent of the `!listIsEmpty` guard (comment at line 108-109: don't let the celebration overlap the empty state) — the fix should still avoid that overlap, just not via an independent, non-animated yank.

Route any timing/animation through `withAppAnimation` (see Sources/Services/TestHooks.swift) as the codebase already does, so the UI tests' animation-disable switchboard still governs it. Do not touch Sources/Views/SparkleBurst.swift — it was investigated and ruled out (its one-shot 0.55s burst completes well before the 1.6s dismiss begins).
