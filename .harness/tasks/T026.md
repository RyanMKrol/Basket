## Overview

Foundation for Siri and widgets: the SwiftData store must live in a shared **App Group** container
so an App Intent and a widget extension can read/write the same list. Enabling the App Group
capability needs the Apple Developer account, so this is a human-gated setup step.

## Do (human-gated — prepare, then a human enables the capability)

1. Choose an App Group id (e.g. `group.com.<team>.basket`) and enable the **App Groups** capability
   for the app target in the Apple Developer portal / Xcode signing (this is the human step —
   provisioning).
2. Declare the App Group entitlement for the app target via `project.yml` (an `entitlements` file
   listing `com.apple.security.application-groups`), and regenerate with `xcodegen generate`. Add
   the required-reason / privacy note if applicable (none new expected for App Groups).
3. **Relocate the SwiftData store** from its current private location (created in `BasketApp.init`)
   to the shared App Group container URL, behind a small helper (e.g.
   `AppGroup.storeURL`) so the app, the future App Intent, and the future widget all resolve the
   same store. Keep the existing `-uiTesting` in-memory store and `UITEST_STORE_URL` temp-store
   paths working unchanged (only the real/default store moves to the App Group).
4. Verify the app still launches, seeds the starter items on first run, and persists across
   relaunches with the store in the shared container.

## Done when

- The app builds and runs with the SwiftData store in the shared App Group container; seed +
  persistence still work (existing `ModelTests` and the relaunch UI tests stay green).
- The `-uiTesting` in-memory and `UITEST_STORE_URL` temp-store test paths are unchanged.
- The App Group id + entitlement are committed (`project.yml` + entitlements file); the human has
  enabled the capability so a real-device/TestFlight build signs. Full simulator suite green.
