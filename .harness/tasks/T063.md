## Overview
One-time environment step every xcui-based check in this cluster depends on. The Axiom plugin ships `xcui` (scriptable simulator UI + accessibility testing), but xcui delegates input and accessibility-tree reading to AXe, a separate Homebrew-installed helper. Installing AXe is a real system-level install on this machine (brew), which is outside the loop's remit, so this is a human step done interactively.

## Do
## Do

1. Run `xcui doctor`. `xcui` is normally on PATH in Claude sessions (the Axiom plugin's bin/ is auto-resolved); from a bare terminal use the plugin binary directly, e.g. `~/.claude/plugins/cache/axiom-marketplace/axiom/<version>/bin/xcui doctor` (pick the newest version directory).
2. If it reports AXe missing (exit 2), run `xcui doctor --install` - this runs `brew install cameroncooke/axe/axe` (explicit, consented). Homebrew must be present.
3. Boot the pinned simulator so doctor can see one: `./build_run.sh` from the repo root boots "iPhone 17 Pro", installs and launches Basket.
4. Re-run `xcui doctor` until it exits 0 (AXe present, brew present, Xcode ok, booted sim found). If doctor reports an `axe_developer_dir` note (Xcode beta relocation), record it in the worklog - bare `axe` calls need `DEVELOPER_DIR=<that value>` prefixed.
5. Sanity-check against the live app: with Basket running from step 3, `xcui assert --id addBar.textField` must pass (the add bar's accessibility identifier, see Sources/Support/A11yIDs.swift), confirming the whole chain (AXe tree read -> xcui assert) works against this app.
6. Mark the task done with `.harness/scripts/mark-done.sh <task-id>` (never hand-edit TASKS.json status).

## Done when
## Done when

- `xcui doctor` exits 0 on this machine with an "iPhone 17 Pro" simulator booted.
- `xcui assert --id addBar.textField` passes against a running Basket launched by `./build_run.sh`.
- Any `axe_developer_dir` caveat doctor reported is noted in the worklog.
- The task is marked done via `.harness/scripts/mark-done.sh`.
