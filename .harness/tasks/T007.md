## Overview

The emoji keyword table's longest-first sort has no tie-break, so equal-length keyword matches
("rice" vs "cake") are decided by an unstable sort.

## Do

`tools/gen_emoji.py` (~lines 305/308) emits the match code in the GENERATED
`Sources/Services/EmojiTable.swift` as `.sorted { $0.0.count > $1.0.count }` (two occurrences:
complex and simple keyword groups). Swift's sort is not guaranteed stable, so "Rice cakes" → 🎂
today could flip to 🍚 on a compiler change. Edit the generator so the emitted sort tie-breaks
deterministically on the keyword itself:
`.sorted { $0.0.count > $1.0.count || ($0.0.count == $1.0.count && $0.0 < $1.0) }`.
Then regenerate: `python3 tools/gen_emoji.py Sources/Services/EmojiTable.swift`. NEVER hand-edit
`EmojiTable.swift` (it is generated; CLAUDE.md → Emoji pipeline).

## Done when

- Both emitted sorts in the regenerated `EmojiTable.swift` carry the keyword tie-break; the diff
  to `EmojiTable.swift` comes from the generator run, not hand edits.
- Native harness passes and the full simulator suite passes (see README.md → Tests). If any
  existing check pins a tie-affected mapping, the now-deterministic result is documented in the
  worklog.
