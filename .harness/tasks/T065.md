## One-off checks without compiling a test (xcui + AXe)

If your change is behind an interaction and NO existing XCUITest exercises it, you don't have to write a throwaway XCUITest just to look at it once - that costs a full `xcodebuild test` cycle. The Axiom plugin's `xcui` (plus the AXe input helper; `xcui doctor` must pass) drives the RUNNING app through the simulator's accessibility tree - the same no-real-input guarantee as XCUITest (input is injected into the simulator, never the real host mouse/keyboard/screen):

    ./build_run.sh                                   # generate + build + install + launch
    UDID=$(xcrun simctl list devices booted | grep -F 'iPhone 17 Pro (' | grep -Eo '[0-9A-Fa-f-]{36}' | head -1)
    xcui wait --for-element itemRow.Milk --timeout 10s
    axe tap --id itemRow.Milk --udid "$UDID"         # tap BY accessibility identifier, never coordinates
    xcui wait --for-element quantityEditor.value
    xcrun simctl io "$UDID" screenshot screenshots/adhoc.png   # or recordVideo for a frame sequence

Rules: interact only by `accessibilityIdentifier` (see `Sources/Support/A11yIDs.swift`); synchronize with `xcui wait` / `xcui assert` between steps, never sleep-and-hope; if `xcui doctor` reports the environment unready, fall back to the temporary-XCUITest + `tools/export_ui_screenshots.sh` path above. And the house rule stands: xcui is for one-off LOOKING - any DURABLE regression check still gets written as a real XCUITest in `UITests/`.
