## Overview
Basket's SwiftData model list ([GroceryItem, KnownItem]) is hand-spelled at four independent container-creation sites in Sources/ today, and the app, the Siri intent, and (soon, task T033) a widget extension all open the SAME App Group store file from different processes. Introduce ONE shared schema constant + container factory in Sources/Support/ and adopt it everywhere, so a future @Model addition cannot leave two processes disagreeing about the schema of one shared SQLite file.

## Do
Create a new file Sources/Support/AppSchema.swift containing (1) the single source of truth for the model set — e.g. `enum AppSchema { static let models: [any PersistentModel.Type] = [GroceryItem.self, KnownItem.self] }` — and (2) a container factory with the three flavors today's call sites need: `makeInMemoryContainer()` (isStoredInMemoryOnly), `makeContainer(url:)` (explicit file URL), and `makeSharedContainer()` (uses `AppGroup.storeURL`; this is the flavor the App Intent — and later the widget — share with the app). The factory functions throw; callers keep their current error handling. Adopt the factory at ALL existing container-creation sites in Sources/, replacing every hand-spelled `GroceryItem.self, KnownItem.self` list: the three branches in `BasketApp.init` (Sources/BasketApp.swift ~26-48: UITEST_STORE_URL temp-file branch -> makeContainer(url:), -uiTesting in-memory branch -> makeInMemoryContainer(), production App Group branch -> makeSharedContainer()), `AddToBasketIntent.resolveContainer` (Sources/Services/AddToBasketIntent.swift ~76-82 — keep the `containerOverride` test seam exactly as is), and the `#Preview` at Sources/Views/ShoppingListView.swift:478 (in-memory flavor). Behavior must be byte-for-byte identical to today: same store URL (`AppGroup.storeURL` -> Basket.store in the App Group container), same ModelConfiguration semantics per branch, and BasketApp's existing fatalError-on-throw stays untouched (a dependent follow-up task adds the recovery path — do NOT add error recovery in this task). Repo rules that bind this task: the App Intents target compiles these same app-target files (AddToBasketIntent.swift), so the factory must compile in that context too; the UI tests rely on the `-uiTesting` (in-memory) and `UITEST_STORE_URL` (temp persistent store, relaunch tests) branches continuing to work through the factory — see TestHooks and BasketApp.init; unit tests must only use the in-memory and temp-URL flavors, never `makeSharedContainer()` (don't touch the real App Group store from a test).

## Done when
A new unit test in Tests/ asserts the factory's model set is identical across container flavors: for makeInMemoryContainer() and makeContainer(url:) (at a FileManager.default.temporaryDirectory URL), the container's schema entity-name set is exactly {GroceryItem, KnownItem} and matches the names derived from AppSchema.models — so a model added to one flavor but not another fails the test. `grep -rn "GroceryItem.self" Sources/` shows the model list spelled in ONE place only (AppSchema.swift); every former call site compiles against the factory instead. The full suite passes — `xcodegen generate && xcodebuild test -project Basket.xcodeproj -scheme Basket -destination 'platform=iOS Simulator,name=iPhone 17 Pro'` — including the existing UITest persistence/relaunch flows that exercise the -uiTesting and UITEST_STORE_URL branches, proving the factory kept them working. README.md's project map documents the new Support/AppSchema.swift file in the same commit.
