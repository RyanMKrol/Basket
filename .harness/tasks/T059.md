## Overview
Two UI-test tooling scripts pipe `xcodebuild test` through `| tail -20`. If the pipe consumer dies or the script is interrupted mid-run, the xcodebuild process can be orphaned and keep running, holding the simulator and racing later builds (which would also confuse the autonomous harness loop's cold-reset step). Replace the pipeline with a redirect-to-log-file pattern: xcodebuild runs as a direct foreground command writing to a log under build/, and the script tails the log afterwards.

Both scripts use `set -euo pipefail` and currently run:

- `tools/record_ui_test.sh` (around lines 70-74):

```sh
xcodebuild test \
  -project Basket.xcodeproj -scheme Basket \
  -destination "platform=iOS Simulator,id=$SIM" \
  -only-testing:"$TEST_ID" \
  | tail -20
```

- `tools/export_ui_screenshots.sh` (around lines 25-30): the same `xcodebuild test ... | tail -20` shape, with `-only-testing:BasketUITests` and `-resultBundlePath "$RESULT_BUNDLE"`.

In EACH script, replace the pipeline with a log-file redirect so xcodebuild is a direct foreground child (never a pipeline member):

```sh
LOG="$PROJECT_DIR/build/<script-specific-name>.log"   # build/ is gitignored
mkdir -p "$PROJECT_DIR/build"
if ! xcodebuild test \
     ... same flags as today ... \
     >"$LOG" 2>&1; then
  echo "xcodebuild test failed. Last 40 lines of $LOG:" >&2
  tail -40 "$LOG" >&2
  exit 1
fi
tail -20 "$LOG"
```

Use a distinct log name per script (e.g. build/record_ui_test.log, build/export_ui_screenshots.log).

Pitfalls the pattern above already accounts for; do not regress them:

1. `set -euo pipefail` semantics: with a bare `xcodebuild ... >"$LOG" 2>&1` line, a test failure would abort the script BEFORE any tail runs, hiding the failure output entirely. The `if ! xcodebuild ...` wrapper keeps set -e from firing while still propagating a nonzero exit (via the explicit `exit 1`). Do not use `xcodebuild ... || true` (that would swallow the failure) and do not reintroduce any pipeline on the xcodebuild line.
2. Exit-code behavior must be preserved exactly: script exits nonzero when the test run fails, zero when it passes. export_ui_screenshots.sh keeps its current fail-fast behavior (test failure means exit 1 before the export step); a later task changes that.
3. `tools/record_ui_test.sh` has an EXIT trap (`cleanup`) that stops the background `simctl io recordVideo` process. The trap must still fire on the new `exit 1` path so the recording is always stopped. Keep the trap and the `RECORD_PID` handling untouched.
4. On success, still print the last 20 lines (the `** TEST SUCCEEDED **` tail), so callers see the same closing output as before.

Do not use em dashes in any echo strings or other copy you add.
