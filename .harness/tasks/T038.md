## Overview
Two related fixes to the 'Clear all' undo-toast flow in Sources/Views/ShoppingListView.swift: (1) shorten the real-world 'Cleared N items — Undo' toast/undo-window duration in Sources/Services/TestHooks.swift from 5.0s to roughly half that, and (2) stop the 'Got it' section briefly flickering back into view for a beat right when that toast auto-expires, caused by a one-frame race between deleting the buffered items from SwiftData and clearing the pendingClearIDs buffer that was hiding them.

## Do
1) In Sources/Services/TestHooks.swift line 45, change `clearToastDuration`'s normal-use branch from `5.0` to `2.5` — i.e. `static var clearToastDuration: TimeInterval { disableAnimations ? 3.0 : 2.5 }`. Leave the `disableAnimations` (UI-testing) branch's `3.0` untouched — it's a separate, already-shortened constant used only under `-uiTestingDisableAnimations` and UI test timing depends on it. Update the doc comment above it (lines 38-44) to say ~2.5s instead of ~5s.
2) Fix the Got-it-section flicker in Sources/Views/ShoppingListView.swift's `commitClear()` (lines 404-412): the current code calls `context.delete(item)` for every buffered id and then, in the SAME synchronous pass, calls `pendingClearIDs.removeAll()`. Because SwiftData's `@Query items` (line 8) does not necessarily refresh synchronously in lockstep with `context.delete`, there's a transient frame where `pendingClearIDs` is already empty (no longer filtering the cleared items out of `recentlyGot`, line 59-62) but `items` still contains the not-yet-actually-removed objects — so the Got-it section flashes back into view for one frame before `items` catches up and it disappears again. Fix by NOT eagerly emptying the whole buffer in `commitClear()`: instead of `pendingClearIDs.removeAll()`, narrow `pendingClearIDs` down to only the ids that are STILL present in `items` after the deletes — e.g. `pendingClearIDs = pendingClearIDs.filter { id in items.contains { $0.persistentModelID == id } }` (or equivalent) — placed in the same spot (after the delete loop, before the toast fade). Since `context.delete` has already been issued, once `items` actually refreshes those ids will no longer be present in `items` and the filter above naturally drops them then too (a later, harmless no-op churn), but on the frame where the flicker used to happen, `pendingClearIDs` still contains the buffered ids and keeps `recentlyGot`'s filter (line 61) excluding them — so the Got-it section never re-appears. `undoClear()` (lines 416-424) is unaffected and keeps its own `pendingClearIDs.removeAll()` as-is (there deletion never happened, so there is nothing stale to filter against). `clearGot()` (lines 383-400) and `clearToken` guard logic are otherwise unchanged.
3) Route any timing/comment references consistently — do not change the toast's visual design in Sources/Views/ClearToast.swift, only ShoppingListView.swift + TestHooks.swift as above.
