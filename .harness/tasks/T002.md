## Overview
The "All done!" celebration overlay (ClearedCelebration, shown when the last to-get item is checked off) appears to snap/clip out of view instead of dissolving smoothly, because its dismiss animation uses the wrong easing curve for a fade-out.

## Do
In Sources/Views/ShoppingListView.swift's `celebrateCleared()` (~lines 386-392), the overlay is shown via `if celebrating && !listIsEmpty { ClearedCelebration(...).transition(.opacity) }` (~line 115-118). Entry already animates correctly (`withAnimation(.easeOut(duration: 0.3)) { celebrating = true }`), but the dismiss 1.6s later uses `withAnimation(.easeIn(duration: 0.4)) { celebrating = false }`. `.easeIn` starts slow and accelerates at the end, so for an opacity-1-to-0 transition the view stays looking almost fully visible for most of the 0.4s and then rapidly drops to invisible in the final fraction of the animation — this is exactly the reported "clips and disappears abruptly" symptom, not a transition that's missing or misattached. Fix by changing that dismiss animation's curve to one whose rate of opacity change is front-loaded or even (so it visibly dissolves throughout the duration rather than holding then snapping): use `.easeOut(duration: 0.5)` (or `.linear(duration: 0.5)` if the builder judges the eased version still not smooth enough on device) in place of `.easeIn(duration: 0.4)`, keeping everything else in `celebrateCleared()` (the 0.3s easeOut entrance, the 1.6s `asyncAfter` hang time, the `guard !celebrating` reentrancy guard) unchanged. Do not add any new @State or modify ClearedCelebration.swift/SparkleBurst.swift — SparkleBurst's own internal 0.55s spark-fade was checked and ruled out as the cause: it finishes and sits invisible well before the 1.6s mark, so it has no interaction with the dismiss fade being fixed here; changing its timing is out of scope for this fix.

## Done when
`celebrateCleared()`'s deferred `withAnimation` call no longer uses `.easeIn`; it uses `.easeOut` or `.linear` with an explicit duration (~0.4-0.6s), so the opacity of the celebration overlay visibly ramps down throughout the animation rather than staying near-opaque and then snapping to invisible in the last moments. Verify by triggering the celebration (check off the last to-get item) via build_run.sh and reviewing the auto-saved screenshot/behavior, or by reasoning about the curve's rate-of-change shape if simulator inspection of a mid-fade frame isn't practical.
