## Overview
Re-attempt of T029 (a view-only Home Screen widget, small + medium, reading the shared App Group store). The first attempt was blocked at rung 0 before any build/test/audit ran: the builder correctly began creating a new WidgetKit extension target's files (BasketWidget/BasketWidget.entitlements, BasketWidget/BasketWidgetExtension.swift) exactly as the spec instructed, but T029's declared scope was only Sources/, project.yml, Tests/, README.md — it never included a path for the new extension target's own directory, which is the natural/required location for any new Xcode target added via XcodeGen. This was a scope-authoring gap, not a difficulty or ambiguity problem, so the fix is adding BasketWidget/ to scope, not changing the spec's substance.

## Do
Add a WidgetKit extension target named BasketWidget to project.yml (regenerate with `xcodegen generate`), with its Swift sources living under BasketWidget/ (entitlements, extension files, TimelineProvider, views, etc. all belong there). The extension shares the App Group entitlement from T026 so it can read the shared SwiftData store. Implement a TimelineProvider that renders, in the app's Pastel Dots look: small — the "N to get" count plus the first couple of items (emoji + name); medium — the count plus a longer top-items list (emoji + name), matching the in-app row style. Reuse the pure Emoji service (plain Swift — add it to the widget target's sources) so items show the same emoji as in-app. Handle the empty state gracefully ("All done" / basket). Cross-process freshness: a widget process NEVER sees the app process's writes automatically — the writing process must nudge WidgetKit. Give the widget a stable kind identifier (e.g. a shared constant `BasketWidgetKind = "BasketWidget"` visible to both the app and widget targets) and add app-side `WidgetCenter.shared.reloadTimelines(ofKind:)` calls (import WidgetKit) at the write choke points in the APP process: (1) in BasketApp's scenePhase `.background` flush (Sources/BasketApp.swift ~70-78, after the `try container.mainContext.save()`) — one place that covers every in-app mutation at the moment the user leaves the app; and (2) in `AddToBasketIntent.perform()` (Sources/Services/AddToBasketIntent.swift, after `try context.save()`) — the headline staleness case: a Siri add while the widget is on screen and the app never launches. Keep a modest fallback timeline policy (e.g. `.after` a few hours) so the widget self-heals if a nudge is ever missed. The widget treats the shared store as READ-ONLY: build its container via the shared schema/container factory in Sources/Support/ (AppSchema + makeSharedContainer(), from the schema-factory task) instead of hand-listing model types, fetch per timeline snapshot, and never insert/save from the widget process. Privacy manifest: the widget extension ships inside the app bundle, so verify every required-reason API the widget code touches is declared in Sources/PrivacyInfo.xcprivacy — App Group UserDefaults reads are already covered (NSPrivacyAccessedAPICategoryUserDefaults, CA92.1); if widget code starts reading file timestamps or any other required-reason API, add the matching entry in the same change, and if project.yml gives the widget target its own resource set, make sure the privacy manifest is included in it. Note: the app-screenshot visual-verify hook does not capture widgets, so rendering is eyeballed by hand in the simulator — record that check in the worklog.

## Done when
The BasketWidget extension target builds as part of `xcodebuild` (a non-compiling widget target fails CI). A hermetic unit test covers the timeline entry computation (given a set of items in a scratch container, produces the expected entry: count, top-N items, emoji, and the empty state) as pure logic with no rendering involved. README documents the widget. The worklog records a manual visual check of both small and medium widget rendering in the simulator. The app-side freshness nudge is verified either by a unit test through a small seam (an injectable reload hook defaulting to WidgetCenter.shared.reloadTimelines, asserted to fire after AddToBasketIntent.perform()'s save) or, if a seam is disproportionate, by a recorded manual check in the worklog: with the widget placed on the simulator home screen, an intent-driven add refreshes it without the app opening. The widget builds its container exclusively through the shared factory (no ModelContainer(for:) call in BasketWidget/), and performs no writes to the store.
