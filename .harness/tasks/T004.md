## Overview

Persistence failures are silently swallowed by `try?`; log them so a failed save/fetch is visible.

## Do

Add `os.Logger` logging (follow the existing pattern in `Sources/Services/TipJar.swift:36`) to the
silent `try?` persistence paths, WITHOUT changing behaviour:
- `Sources/BasketApp.swift:54` — the background `try? container.mainContext.save()`.
- `Sources/BasketApp.swift:74` — `(try? context.fetchCount(...)) ?? 0`; note a failed fetch here
  would re-seed a non-empty store, so log at error level when the fetch itself throws.
- `Sources/BasketApp.swift:82` — the seed `try? context.save()`.
- `Sources/Services/KnownItems.swift:11` — the upsert fetch.
Convert each `try?` to a `do/catch` that logs the error (subsystem = the bundle id, a sensible
category like "persistence") and otherwise preserves the exact same fallback behaviour.

## Done when

- No behaviour change: the four sites keep their current fallback semantics (skip save / count 0 /
  skip seed-save / treat as no-existing-row), but each failure path emits a `Logger` error line.
- No new force-try (`try!`) anywhere; the app still builds and the full test suite passes
  (`xcodegen generate && xcodebuild test -project Basket.xcodeproj -scheme Basket -destination
  'platform=iOS Simulator,name=iPhone 17 Pro'`).
