## Overview
Owner-approved follow-up to the 'salad' emoji fix: the semantic-fallback anchor list (Sources/Services/SemanticEmoji.swift) has several condiment/canned-goods anchors (vinegar, sauce, ketchup, mustard) that all share the 🥫 glyph, which is what pulled 'salad' toward the wrong glyph before it had its own explicit keyword. Other words that currently fall through EmojiTable to the semantic stage may be landing on similarly-wrong anchors for the same reason. This task adds tooling to surface every such case for human review — it does NOT itself decide which mappings (if any) are wrong or change any anchor/threshold.

## Do
Extend tools/audit_coverage.swift with a new report mode (e.g. a `-semantic-report` flag) that, for every word in tools/corpus/*.txt which falls through EmojiTable.match to SemanticEmoji.match, records: the input word, the nearest anchor word it matched, the similarity score, and the resulting glyph. Group and print the output by resulting glyph, so glyphs claimed by multiple, semantically-unrelated anchor words (like 🥫's vinegar/sauce/ketchup/mustard cluster) are easy to spot in the grouped listing. Write this report to a plain text file (e.g. tools/semantic-anchor-report.txt, gitignored — this is a diagnostic artifact, not curated data) as well as printing a summary to stdout. Do NOT change SemanticEmoji.swift's anchor list, threshold constant, or EmojiTable.swift/gen_emoji.py in this task — this is read-only investigation tooling; deciding whether any specific collision is actually wrong, and fixing it, is explicitly left to the owner reviewing the report and a subsequent task, not this one.

## Done when
Running `swift tools/audit_coverage.swift -semantic-report` (or however the new flag is wired, matching the file's existing flag-handling pattern like `-correctness`) produces tools/semantic-anchor-report.txt listing every corpus word that falls through to the semantic stage, grouped by resulting glyph, each entry showing the matched anchor and similarity score. The report clearly surfaces any glyph claimed by multiple distinct anchor words (at minimum, confirm the report surfaces the known vinegar/sauce/ketchup/mustard -> 🥫 cluster as one such group, proving the tool works). The existing default and `-correctness` audit modes are unaffected (same pass/fail behavior as before this change). No changes to Sources/Services/SemanticEmoji.swift, Sources/Services/EmojiTable.swift, or tools/gen_emoji.py.
