## Overview
Basket deliberately engineers its VoiceOver structure: each item row collapses to ONE announcement stop (name, button trait, restore hint) SEPARATE from its check-button stop (see Sources/Views/ItemRow.swift ~lines 90-110), and AddBar/QuantityEditor carry curated labels and hints. But UITests/AccessibilityAuditTests.swift's `performAccessibilityAudit` cannot assert focus ORDER or announcement PHRASING - a regression that scrambles row reading order or merges stops passes every audit today. Add a local regression check built on the Axiom plugin's `xcui voiceover traverse` / `xcui voiceover assert`, which compute the VoiceOver announcement sequence from the simulator's accessibility tree (no XCUITest compile, no captured audio) and diff it against a committed expected sequence.

## Do

Create `tools/verify_voiceover_order.sh` (bash, `set -euo pipefail`, executable), modeled on `tools/record_ui_test.sh`'s simulator handling:

1. **Resolve the simulator**: default name "iPhone 17 Pro" (optional first arg overrides), resolved to a concrete UDID exactly like record_ui_test.sh does (`xcrun simctl list devices booted` first, then `available`).
2. **Preflight**: run `xcui doctor` and fail fast (clear message, nonzero exit) if it exits nonzero - do not limp on without AXe. `xcui` is the Axiom plugin binary, normally on PATH; if `command -v xcui` fails, fall back to the newest `~/.claude/plugins/cache/axiom-marketplace/axiom/*/bin/xcui`.
3. **Build + install + launch**: run `./build_run.sh "$SIM_NAME"` (generates, builds, installs, launches). Then relaunch DETERMINISTICALLY: `xcrun simctl terminate "$UDID" com.ryankrol.basket || true`, then `SIMCTL_CHILD_UITEST_FROZEN_DATE=2026-07-15T09:00:00Z SIMCTL_CHILD_TZ=Europe/London xcrun simctl launch "$UDID" com.ryankrol.basket -uiTesting -uiTestingDisableAnimations`. This mirrors UITests/BasketUITestCase.swift exactly: isolated in-memory SwiftData store seeded with the four starter items (Milk, Sourdough bread, Eggs, Tomatoes), animations off, clock frozen at the same instant and zone the UI tests pin - so the computed sequence is stable run to run (see Sources/Services/TestHooks.swift).
4. **Synchronize, never sleep**: `xcui wait --for-element itemRow.Milk --timeout 15s --udid "$UDID"` before reading the tree (bounded wait on an accessibilityIdentifier, matching the repo's UI-test house rule; identifiers live in Sources/Support/A11yIDs.swift).
5. **Default mode**: run `xcui voiceover assert --sequence tools/voiceover_expected.json --udid "$UDID"` and exit with its exit code (0 = match; 1 = mismatch, with per-index diffs in the JSON output - let that output reach stdout).
6. **`--record` flag**: instead write `xcui voiceover traverse --udid "$UDID"` output to `tools/voiceover_expected.json` (the assert verb round-trips a saved traverse report), for intentional announcement changes.
7. **Record and COMMIT the initial expected file** via `--record`. Before committing, eyeball it: each starter item must appear as ONE info stop labelled with the item name (button trait, "Double tap to restore"-style hint) plus a SEPARATE check-button stop ("Not got yet"), in top-to-bottom order with the header count, about button, and add bar where the layout puts them. If rows read as multiple stops or merged stops, that is an app bug to report, not an expected file to bless.
8. **Prove it can fail**: temporarily perturb a COPY of the expected sequence (swap two entries), confirm the script exits nonzero with xcui's per-index diff, restore. Do not commit the perturbation; note the check in the worklog.
9. **Document in README.md** (same commit, in the Tests/tools area): what the check asserts, how to run it, and that `--record` + a justified diff is the path for intentional announcement changes.

Hard constraints: this drives the SIMULATOR only, via simctl and the accessibility tree - never the real host mouse/keyboard/screen (no cliclick/AppleScript/CGEvent; see README's UI-test section). Do NOT touch `.github/workflows/ci.yml` (CI runners do not have the Axiom plugin binary - this is a local, on-demand check), and do NOT touch UITests/, Sources/, or the xcodebuild scheme. No user-facing app copy changes, so the no-em-dash copy rule is not in play, but keep shell output plain ASCII.
