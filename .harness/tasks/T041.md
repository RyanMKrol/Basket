## Overview
The UI-test base class's own house rule says never assert live UI state directly ŌĆö every assertion goes through a bounded wait ŌĆö yet six call sites assert element ABSENCE as an instantaneous `XCTAssertFalse(element.exists)` read. Absence right now does not mean absence after the UI settles: the worst case, UITests/DeepLinkTests.swift (testNormalLaunchDoesNotAutoFocus), asserts the keyboard-dismiss button doesn't exist immediately after the add-bar field appears ŌĆö if auto-focus wrongly arrived a beat later, the test would falsely PASS. This task adds a bounded stays-absent assertion helper, `assertStaysGone`, to `UITests/BasketUITestCase.swift` (poll over a short settle window, failing fast the moment the element appears; NEVER `sleep()`), routes every identified `XCTAssertFalse(...exists)` site through it, and updates the base class's synchronization doc comment so the house rule covers absence as well as presence.
