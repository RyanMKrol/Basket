## Overview
When typing an exact quantity into the Quantity Editor's text field there's no keyboard-side way to dismiss it (deliberately no on-keyboard Done button per the comment at lines 76-78, since it overlapped the list); add Return/Enter as a way to resign focus and commit the typed value, without closing the surrounding editor.

## Do
On the `TextField` in `QuantityEditor.valueDisplay` (~lines 60-82), add `.submitLabel(.done)` (labels the keyboard's return key "Done") and `.onSubmit { fieldFocused = false }`. Do not call `commit()` directly from `onSubmit` — setting `fieldFocused = false` alone is sufficient, because the existing `.onChange(of: fieldFocused)` handler at lines 79-81 already calls `commit()` whenever focus leaves, so this reuses the single existing commit path instead of duplicating it (avoids double-invoking `onSetValue`). This change only touches `fieldFocused`/`editing` state inside `QuantityEditor` — it must NOT touch `ShoppingListView`'s `expandedID`, so the stepper/unit-pills row stays open/visible after Return; only the keyboard goes away and the value commits.

## Done when
With an item's quantity editor open and the amount tapped into edit mode (keyboard visible), typing a new number and pressing the keyboard's Return/Done key dismisses the keyboard and commits the typed value (the item's `quantity` updates via `onSetValue`, same as tapping away does today), while the quantity editor row (stepper + unit pills) remains visibly expanded — i.e. `expandedID` in `ShoppingListView` is untouched by this change. Verify manually via `./build_run.sh` or the Xcode simulator (tap the qty amount, type a number, press Return, confirm keyboard dismisses + value updates + editor stays open) since this is keyboard/focus UI behavior outside the native logic harness/XCTest scope.
